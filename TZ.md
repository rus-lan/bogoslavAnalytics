## 1. Цель и сценарий использования

### 1.1. Цель

Инструмент аналитики ревью-активности в GitLab. Основной вопрос, на который он отвечает: **в каких merge requests конкретный пользователь оставил больше N комментариев** (N — динамический параметр), с фильтрами по дате, группе и репозиторию. Дальше по цепочке: вытащить сами комментарии, разметить их по логическому смыслу (силами вызывающего агента), отфильтровать по смысловым группам. Каждый шаг пишет статический файл (text / yaml / json), который одновременно является артефактом-результатом и кешем для следующих шагов.

### 1.2. Сценарий использования

1. Пользователь в агентском инструменте (например, opencode) вызывает скилл.
2. Скилл ведёт агента по конвейеру: агент вызывает тулы MCP-сервера `bogoslav-mcp` (или напрямую команды `bogoslav-cli` — поверхность одна и та же).
3. MCP-сервер ходит в GitLab API, применяет стратегию поиска, пишет артефакты.
4. Семантическую разметку комментариев выполняет сам вызывающий агент (LLM), MCP лишь выдаёт батч + таксономию + JSON-схему и валидирует результат.
5. Повторные запросы с теми же параметрами обслуживаются из артефактов-кешей без похода в API.

Поддерживается точечная обработка: конкретный MR (`--project` + `--mr`), конкретная группа (`--group`), конкретный репозиторий (`--project`) — без обязательного прохода всего конвейера.

### 1.3. Роли компонентов

| Компонент | Что делает | Чего НЕ делает |
|---|---|---|
| `bogoslav-cli` | Детерминированные шаги: поиск MR, выгрузка комментариев, подготовка батча разметки, приём/валидация разметки, фильтрация, статистика | Не вызывает LLM |
| `bogoslav-mcp` | Те же операции как MCP-тулы поверх тех же internal-пакетов | Не вызывает LLM, не размечает |
| `bogoslav-skills` | Генерирует SKILL.md из дерева команд cobra и устанавливает скиллы + MCP-конфиги в целевые инструменты | Не содержит бизнес-логики |
| Вызывающий агент | Семантическая разметка комментариев по таксономии и JSON-схеме | Не ходит в GitLab напрямую |

## 2. Архитектура и границы

### 2.0. Целевая версия GitLab: 18.11 self-managed

Интеграция нацелена на **GitLab 18.11 self-managed**. Факты ниже сверены с `archives.docs.gitlab.com/18.11/` и веткой исходников `18-11-stable-ee`; там, где остальной текст ТЗ (разделы 5–6) написан под GitLab вообще, это — конкретика для этой цели, не новая, отдельная от него платформа.

1. **Заголовки лимита скорее всего ОТСУТСТВУЮТ на целевом инстансе.** На self-managed троттлинг аутентифицированного API по умолчанию ВЫКЛЮЧЕН, а `18-11-stable-ee` строит заголовки `RateLimit-*` из данных совпадения Rack::Attack: если ни одно правило троттлинга не сработало, заголовки не эмитятся вообще (не как нули — их попросту нет). Админ-документация 18.11 утверждает, что заголовки «включены во все ответы» — для ненастроенного инстанса это неверно. Следствие: наш лимитер, управляемый по каждому ответу (раздел 6), при отсутствии заголовков падает на экспоненциальный бэкофф — это и есть верное поведение для этой цели. Лимитер, зашитый на путь эндпоинта, на этом инстансе никогда бы не адаптировался — фиксируем явно, чтобы никто не «починил» это в путевой лимитер позже.
2. **Retention событий зависит от инстанса — это НЕ гарантированный 3-летний пол.** `PruneOldEventsWorker` на 18.11: `CUTOFF_DATE = 3.years + 1.day`, `DELETE_LIMIT = 10_000` за один прогон, cron `0 */6 * * *`, включён ops-флагом `ops_prune_old_events` (`default_enabled: true`, флаг добавлен в 15.10, MR !115100). Флаг может выключить только администратор через Rails-консоль; UI-настройки нет, менять окно в 3 года нельзя. На загруженном инстансе лимит 10k/6ч означает, что события старше порога могут временно пережить удаление. Три года — не гарантированный пол (флаг можно выключить) и не жёсткий потолок (удаление может отставать) — подробности и следствия для автоселектора см. раздел 5.3.
3. **`non_archived` имеет РАЗНЫЕ дефолты по эндпоинтам на 18.11**: список MR группы (`GET /groups/:id/merge_requests`) — дефолт `true`; глобальный список (`GET /merge_requests`) — дефолт `false`. Мы отправляем `non_archived=false` ЯВНО на обоих (раздел 5.2, п. 7) — иначе bruteforce по группе тихо теряет MR из заархивированных проектов там, где events их всё равно находит через комментарии: расхождение стратегий на одном и том же запросе.
4. **Параметра-фильтра `draft` для списков MR не существует до GitLab 19.0** — на 18.11 эквивалент называется `wip` (`yes`/`no`). Мы не фильтруем черновики вообще ни тем, ни другим способом; параметр `wip` не использовать без отдельного решения.
5. **Устаревшие, но ещё присутствующие на 18.11 атрибуты — не полагаться:** `merge_status` (deprecated 15.6, использовать `detailed_merge_status`), `merged_by` (deprecated 14.7, использовать `merge_user`), `work_in_progress` (устаревшее поле ОТВЕТА, использовать поле ответа `draft` — оно на 18.11 уже есть и не связано с фильтром-параметром из п. 4 выше), `reference` (deprecated 12.7, использовать `references`). Кодовая база сегодня ни одно из этих полей не читает и не отправляет — фиксируем как «не заводить» на будущее, а не как найденный дефект.
6. **Авторизация**: на 18.11 работают оба варианта — заголовок `PRIVATE-TOKEN` и `Authorization: Bearer <token>` для personal access token (раздел 2.5 уже фиксирует это как решение; здесь — подтверждение, что оба пути верны именно для целевой версии).
7. **Forward-compat, не наша проблема на 18.11, но фиксируется на будущее**: GitLab 19.3 делает REST API возвращающим `400` для нечисловых ID в целочисленных path-параметрах — если инструмент когда-нибудь будет таргетить 19.3+, часть `:id`-путей раздела 5.7/14 (п. 1) потребует пересмотра.

### 2.5. Конфигурация подключения

- URL инстанса: переменная окружения `GITLAB_URL` (для gitlab.com можно опустить — тогда дефолт `https://gitlab.com`, `gitlab.NewClientFromEnv`).

## 4. Конвейер артефактов и схемы

Четыре артефакта. Каждый несёт общий заголовок (Go-тип `artifact.Header{SchemaVersion, Kind, Source}`, встроенный в тип каждого документа), поле `schema_version` (целое, `artifact.CurrentSchemaVersion = 1`), поле `kind`, блок провенанса `source`. Форматы — **четыре**, не три: `text`\|`yaml`\|`json`\|`html` (флаг `--format`, дефолт `yaml`). Каждый артефакт — одновременно кеш: перед походом в API система ищет готовый артефакт с тем же нормализованным запросом и не истёкшим TTL.

`text` и `html` — оба **write-only** и относятся к одному правилу: `artifact.Format.writable()` возвращает true только для `json`/`yaml`. Ни один из двух не кешируется (поиск кеша строит только кандидатов `.yaml`/`.json` — раздел 4.6 — так что `.txt`/`.html` структурно исключены, а не отфильтрованы постфактум) и не принимается как вход `from_artifact` ни в одном шаге конвейера; попытка прочитать `.txt`/`.html` завершается ошибкой `artifact.ErrNotReadable` (переименован из более раннего `ErrTextNotReadable` — правило одно на оба формата). `FormatFromPath` также распознаёт расширение `.htm` как `FormatHTML`.

Причина для `text`: он несёт человекочитаемую шапку-комментарий и не раунд-трипится обратно в структуру. Причина для `html` та же плюс собственная: html — самостоятельная презентационная страница, не структура данных.

#### 4.0. Формат `html`

Каждый артефакт умеет рендериться как один самодостаточный HTML-файл: инлайновые `<style>` (без внешнего CSS/шрифтов/скриптов, без `@import`), светлая/тёмная тема через `prefers-color-scheme`, широкие таблицы прокручиваются в собственном контейнере (`.table-scroll { overflow-x: auto }`), а не ломают страницу. Файл рассчитан на офлайн-просмотр (вложение в тикет, письмо).

Рендер построен на `html/template` (стандартная библиотека) — это требование безопасности, а не стиля: тела комментариев (`Note.Body`) и `web_url` — недоверенный текст пользователей GitLab. `html/template` делает контекстное автоэкранирование: `{{.Body}}` внутри `<div class="body">` рендерится экранированным (`<script>` → `&lt;script&gt;...`, проверено тестом на XSS-пейлоады), а `{{.WebURL}}` внутри `href="..."` — в URL-контексте: значение со схемой `javascript:` не проходит и заменяется на `href="#ZgotmplZ"` (документированное поведение `html/template` для содержимого, которое пакет считает небезопасным в URL-позиции). Markdown в теле комментария намеренно НЕ рендерится (потребовал бы отдельного рендерера плюс санитайзера сверх него) — тело печатается как есть, экранированным, со стилем `white-space: pre-wrap`.

В `labeled_comments` (и, соответственно, в шаблоне, общем с `filtered_comments`) HTML-вид показывает провенанс-блок `classifier` в отдельной, явно выделенной секции (`class="provenance"`), которая рендерится ДО секции с содержимым (`class="content"`) — раньше, чем сами размеченные комментарии, а не среди них.

### 4.1. Общая шапка (`Header`) и формы `text`/`json`/`yaml`

Go-тип шапки:

```go
type Header struct {
    SchemaVersion int    `json:"schema_version"`
    Kind          Kind   `json:"kind"`
    Source        Source `json:"source"`
}
type Source struct {
    GitlabURL string    `json:"gitlab_url"`
    FetchedAt time.Time `json:"fetched_at"`
}
```

yaml-вид (в json — те же ключи; в text — человекочитаемая шапка-комментарий):

```yaml
schema_version: 1
kind: mr_list            # mr_list | comment_list | labeled_comments | filtered_comments
source:
  gitlab_url: https://gitlab.example.com
  fetched_at: 2026-07-15T12:00:00Z
query:                   # нормализованный запрос, породивший артефакт
  ...
```

`source.fetched_at` не было названо явно в исходной редакции этого раздела; на деле оно есть и является load-bearing полем: свежесть кеша (раздел 4.6) определяется сравнением `fetched_at` с TTL, а не временем модификации файла на диске.

### 4.2. Артефакт-1 `mr_list` — найденные MR со счётчиками (детерминированный шаг)

```yaml
kind: mr_list
query:
  gitlab_url: https://gitlab.example.com   # входит в нормализованный запрос — раздел 4.6
  user_id: 42
  from: "2026-01-01"
  to: "2026-06-30"
  group: "my-group"          # опционально
  project: "my-group/repo"   # опционально
  mr: 77                      # опционально — точечный режим, раздел 7.2
  more_than: 5                # порог N — comment_count СТРОГО больше N
  strategy: events           # events | bruteforce — что реально применилось
  smoke: passed                # passed | failed | unknown — результат smoke-теста (раздел 5.5)
items:
  - project_id: 123
    project_path: "my-group/repo"
    mr_iid: 77
    comment_count: 8         # точное число комментариев юзера в окне
    title: "fix bug"          # опционально, omitempty
    web_url: "https://gitlab.example.com/my-group/repo/-/merge_requests/77"   # опционально, omitempty
    created_at: "2026-03-01T10:00:00Z"   # опционально, omitzero
    updated_at: "2026-03-05T10:00:00Z"   # опционально, omitzero
```

`comment_count` — всегда точный подсчёт по `/discussions` (раздел 5.4), независимо от стратегии поиска кандидатов. Итоговый предикат — строго `comment_count > more_than` (не `>=`): кандидатные пре-фильтры (§5.1.4, §5.2.3) используют `>=`/`<` как надмножественную (superset) оценку кандидатов, но границу пиннит только это точное сравнение.

Строка `items[]` — Go-тип `artifact.MRItem`. `title`, `web_url`, `created_at`, `updated_at` — необязательные поля (`omitempty`/`omitzero`), которых не было в исходной иллюстрации этого раздела: они добавлены аддитивно, чтобы HTML-вид (раздел 4.0) мог связать заголовок MR со ссылкой и показать даты. Писатель без этих данных просто оставляет их нулевыми, и они не попадают на wire; на JSON Schema/OpenAPI-контракт (раздел 10) их отсутствие не влияет — они всегда необязательны.

Статус реализации — обновлено этой ревизией, один пробел закрыт на уровне `domain`/`gitlab`, второй остаётся открытым на уровне `artifact` (раздел 14, п. 10):

- **Расхождение формы MR между стратегиями — ЗАКРЫТО.** Новая функция `search/enrich.go`'s `enrichEventsCandidates` дозаполняет `Title`/`WebURL`/`CreatedAt`/`UpdatedAt`/`ProjectPath` у выживших кандидатов events-стратегии (после точного `> MoreThan`-фильтра, раздел 5.7) через `gitlab.ProjectMergeRequestsByIIDs` — по одному вызову на проект. Теперь `search.Find` для обеих стратегий отдаёт `[]domain.MergeRequest` одной и той же формы; тест-регрессия — `search.TestFind_eventsAndBruteforceStrategiesProduceEquivalentMergeRequests`.
- **`domain.MergeRequest.ProjectPath` — теперь ИМЕЕТ продюсера.** Поле (`json:"project_path,omitempty"`) и новый саб-тип `domain.References{Short, Relative, Full}` с методом `References.ProjectPath()` (делит `Full` по ПОСЛЕДНЕМУ `!`, поскольку `!` не может входить в путь GitLab) заполняются функцией `gitlab.populateProjectPaths` из `references.full` на каждом MR-list-эндпоинте (`MergeRequests`, `GroupMergeRequests`, `ProjectMergeRequests`, `ProjectMergeRequestsByIIDs`) — без единого лишнего запроса к API. Тест: `gitlab.TestClient_MergeRequestLists_populateProjectPathFromReferences`.
- **`artifact.MRItem.ProjectPath` (поле артефакта-1, ниже) — продюсера ВСЁ ЕЩЁ НЕТ.** Данные для него теперь готовы на уровне `domain.MergeRequest` (пункт выше), но ничто в коде продукта не конвертирует `search.Result` в `artifact.MRList`/`artifact.MRItem` — это тот же незакрытый пробел, что и раньше (раздел 14, п. 10), просто теперь конвертеру, когда он появится, будет из чего брать `project_path` без дополнительных походов в API. Поле `artifact.MRItem.ProjectPath` несёт тег `omitempty` (это уже так в коде — более ранняя редакция этого раздела ошибочно утверждала обратное).

### 4.3. Артефакт-2 `comment_list` — вытащенные комментарии (детерминированный шаг)

Реальные wire-имена полей строки `items[]` — это поля переиспользуемого доменного типа `domain.Note` (см. раздел 2.3), а не отдельный «плоский» тип с косметическими именами вроде `note_id`/`author_id`/`text`: дублировать доменный тип ради других имён полей было осознанно отклонено. Настоящая форма:

```yaml
kind: comment_list
query:
  user_id: 42
  from: "2026-01-01"
  to: "2026-06-30"
  mrs:                       # либо ссылка на артефакт-1, либо явный список
    - {project_id: 123, mr_iid: 77}
  from_artifact: "artifacts/mr_list_<hash>.yaml"   # если строился из артефакта-1
items:
  - mr_iid: 77                # iid MR, к которому относится комментарий (не noteable_id заметки)
    project_id: 123
    id: 456                   # id заметки (было бы "note_id" в косметической схеме)
    type: DiscussionNote        # DiscussionNote | DiffNote | null (domain.NoteType, раздел 2.3)
    body: "..."                # текст комментария (было бы "text")
    author:
      id: 42                   # numeric id автора (было бы "author_id")
      username: "alice"
    created_at: "2026-03-01T10:00:00Z"
    updated_at: "2026-03-01T10:05:00Z"   # опционально, omitzero
    system: false
    noteable_id: 999
    noteable_type: "MergeRequest"
    resolved: false            # опционально, omitempty
    resolvable: false          # опционально, omitempty
```

Строка `items[]` — Go-тип `artifact.CommentItem{MRIID int64; domain.Note}` (embedding): `mr_iid` хранится рядом с заметкой отдельно, поскольку `NoteableID` самой заметки — это внутренний id MR, а не его iid. Форма ответа `GET /projects/:id/merge_requests/:iid/discussions`, из которой вычитывается `domain.Note`, — верифицирована, фактам ниже не хедж:

- Объект discussion: `id` (**строка**, SHA-подобная), `individual_note` (boolean), `notes` (массив).
- Поля объекта note: `id` (**целое**), `type` (строка: `DiscussionNote` | `DiffNote` | `null`), `body` (строка, текст комментария), `author` (объект), `created_at` (строка ISO 8601), `updated_at`, `system` (boolean), `noteable_id` (целое), `noteable_type` (строка), `project_id` (целое), `resolved`, `resolvable`, `resolved_by`, `resolved_at`, `position` (объект, только у diff-заметок), `suggestions` (массив).
- `notes[].author.id` — верный путь к numeric ID автора. `notes[].system` — верный флаг системной заметки. `notes[].created_at` присутствует.
- Записи типа `DiffNote` дополнительно несут `commit_id` и объект `position` (поля `base_sha`, `start_sha`, `head_sha`, `old_path`, `new_path`, `position_type`, `old_line`, `new_line`, `line_range`).
- **Не путать**: `id` дискуссии — строка, `id` заметки внутри `notes[]` — целое число.
- Оговорка: пример JSON в доках датирован эрой GitLab 10.x; современные инстансы могут возвращать доп. недокументированные поля заметок. Контракт — таблица документированных атрибутов; парсить нестрого (неизвестные поля игнорировать).

### 4.4. Артефакт-3 `labeled_comments` — комментарии + смысловые метки + провенанс (шаг агента/LLM)

Всё содержимое артефакта-2 плюс таксономия и провенанс. Строка `items[]` — Go-тип `artifact.LabeledCommentItem{MRIID int64; domain.LabeledNote}`, где `domain.LabeledNote` встраивает `domain.Note` и добавляет `label` (те же реальные wire-имена `id`/`author.id`/`body`, что в артефакте-2 — раздел 4.3):

```yaml
kind: labeled_comments
taxonomy:                    # копия применённой таксономии
  version: 3
  labels: [...]
classifier:                  # провенанс разметки — ОБЯЗАТЕЛЕН
  tool: opencode
  model: glm-5.2
  taxonomy_version: 3
  classified_at: "2026-07-15T16:40:00Z"
items:
  - mr_iid: 77
    project_id: 123
    id: 456                   # id заметки (не "note_id")
    body: "..."                # текст комментария (не "text")
    author:
      id: 42
      username: "alice"
    created_at: "2026-03-01T10:00:00Z"
    ...
    label: "style"           # метка из таксономии
```

Артефакт-3 пишется **только** тулом `save_labels` (или командой `bogoslav-cli classify save`) после успешной валидации разметки по JSON-схеме. Разметка, не прошедшая схему, в артефакт не попадает — операция завершается ошибкой, файл не создаётся и не перезаписывается. Провенанс-блок `classifier` обязателен целиком: запись пустого/неполного блока (`tool`, `model`, `taxonomy_version <= 0`, `classified_at` нулевой) завершается ошибкой `artifact.ErrMissingClassifier`, артефакт-3 не создаётся вообще (ни один из четырёх форматов).

### 4.5. Артефакт-4 `filtered_comments` — результат фильтрации по группам (детерминированный шаг)

kind: filtered_comments

### 4.6. Роль кеша и переиспользование

- Имя файла артефакта: `<kind>_<hash>.<ext>` (`cache.FileName`), где `hash` — **SHA-256** от канонического JSON-представления нормализованного запроса (`cache.Hash`/`cache.QueryHash`): ключи объекта отсортированы лексикографически на каждом уровне вложенности, числа проходят через `json.Number` (без потери точности через `float64`), даты приведены к `YYYY-MM-DD` (`domain.Date.String()`), опущенные необязательные параметры **отсутствуют как ключи** (не сериализуются как `null`). В хешируемый объект всегда входят `gitlab_url`, резолвленный numeric `user_id`, `from`, `to`, `more_than`; `group`/`project`/`mr` — только если заданы. Нормализацию и хеш реализует `cache/`; представление стабильно между процессами (не зависит от порядка вставки в map — Go's `encoding/json` сортирует ключи `map[string]any` при маршалинге).
  - Хешируемый объект **намеренно исключает** `Strategy` и `Smoke`, хотя оба — поля `domain.Query` и попадают в блок `query` самого артефакта для провенанса (разделы 4.2, 5.3, 5.5): оба решаются ВО ВРЕМЯ выполнения шага, а поиск в кеше происходит ДО его выполнения — хеширование их сделало бы ключ поиска зависимым от результата, которого ещё не существует, и от того, смотрит ли вызывающий код в кеш до или после того, как стратегия определилась.
  - Правило «опущено vs явно задано»: `Group`/`Project` в `domain.Query` — обычные строки, поэтому опущенное значение и явная пустая строка — один и тот же нулевой Go-тип и хешируются ОДИНАКОВО (обе — «ключ отсутствует», как и `omitempty`-тег этих полей на самом `domain.Query`). `MR`, напротив, — `*int64`: `nil` (опущено) и указатель на `0` (явно задан `mr=0`) различимы и хешируются ПО-РАЗНОМУ.
  - Статус реализации / опасность, ПЕРЕФОРМУЛИРОВАНА под новую реальность (раздел 14, п. 9): конвертера `domain.Query` → `search.Params` всё ещё не существует, но сама природа опасности изменилась. Раньше `search.Params.Scope` принимал уже резолвленные ЧИСЛОВЫЕ id, а `QueryHash` хешировал строковые пути — рассинхрон между ключом (строка) и областью поиска (число) был реальной дырой. Теперь `Scope{GroupID, ProjectID *gitlab.ID}` (раздел 5.7) принимает путь НАПРЯМУЮ — путь и есть то, чем реально ходят в API (раздел 14, п. 1) — так что ключ кеша (по строке пути) снова совпадает с тем, что реально запрашивается: старое рассогласование «путь vs numeric id» закрыто. Более УЗКАЯ опасность остаётся: если группу/проект переименуют и НОВЫЙ объект займёт СТАРЫЙ путь, тот же ключ кеша будет отвечать за другие данные — но это ограничено TTL в 24 часа (раздел 4.6 ниже), и принимается как оправданный риск, а не как нерешённая проблема.
- Перед выполнением шага система ищет в `--out`-каталоге артефакт с тем же `kind` и хешем. `cache.Lookup` строит кандидатов только с расширениями `.yaml`/`.json` (`.txt`/`.html` структурно исключены, не отфильтрованы постфактум — раздел 4). Если найден и `fetched_at` моложе TTL — шаг **не ходит в API**, отдаёт путь к существующему файлу. `cache/` не импортирует `artifact/` (раздел 2.4): за `fetched_at` кеш обращается через собственный потребительский интерфейс `cache.HeaderReader{FetchedAt(path) (time.Time, error)}`, а не разбирает форматы сам.
  - Статус реализации (раздел 14, п. 8) — ЗАКРЫТО на уровне компонента: `cache.HeaderReader` теперь имеет боевую реализацию, `artifact.HeaderStore.FetchedAt(path string) (time.Time, error)` (`artifact/header_store.go`), поверх того же `decodeFile`, которым читаются четыре `ReadX`-функции; компайл-тайм доказательство соответствия интерфейсу — `var _ cache.HeaderReader = (*HeaderStore)(nil)` в `artifact/header_store_test.go`, без цикла импортов (`cache/` по-прежнему не импортирует `artifact/`). Тесты: `TestFetchedAt_roundTripsAcrossFormatsAndKinds`, `_txtIsNotReadable`, `_htmlIsNotReadable`, `_missingFileIsWrappedError`, `_malformedFileIsWrappedError`, `_unknownSchemaVersionIsWrappedError`. Нюанс, который нельзя терять: `cache.Lookup` сегодня всё ещё не вызывается НИКАКИМ кодом продукта — не потому, что `HeaderReader` нечем удовлетворить, а потому, что `bogoslav-cli`/`bogoslav-mcp` остаются заглушками (раздел 7) и никто пока не собирает `cache.Lookup` + `artifact.HeaderStore` в одну цепочку вызовов.
- TTL: дефолт 24h (`cache.DefaultTTL`), флаг `--cache-ttl`. Флаг `--refresh` принудительно игнорирует кеш (`cache.Options.Refresh` — форсирует промах без обращения к файловой системе).
- Переиспользование по конвейеру: артефакт-2 строится из артефакта-1 (список MR берётся из файла, в API идут только запросы `/discussions` по перечисленным MR); артефакт-3 — из артефакта-2 (в API не ходит вообще); артефакт-4 — из артефакта-3 (в API не ходит вообще). Шаги 3 и 4 при валидном входном артефакте не делают ни одного HTTP-запроса.
- Кеш разметки (артефакт-3): ключ `cache.LabelKey{ItemsHash, Model, TaxonomyVersion}` (`cache.NewLabelKey`) — если содержимое комментариев батча не изменилось (`ItemsHash` — хеш по `items` артефакта-2) и пара `Model` + `TaxonomyVersion` совпадает с уже существующим артефактом-3 — повторная разметка не запускается, `get_classify_batch` возвращает признак `cached: true` и путь к готовому артефакту-3. `LabelKey` сравним через `==`.

## 5. Стратегии поиска

Гибрид: основная стратегия — **events**, фолбэк — **bruteforce**. Стратегию выбирает **автоселектор**, не пользователь (у пользователя есть только флаг `--strict`, форсирующий bruteforce). Оценка стоимости из research: на годовом окне для активного ревьюера гибрид — ~250–400 запросов, перебор по группе — ~3400.

### 5.0. Резолюция `--user`: username → numeric ID

Флаг `--user` (и одноимённый вход тулов) принимает **и** numeric ID, **и** username:

- Значение из одних цифр — трактуется как numeric ID напрямую.
- Иначе — резолвится один раз через `GET /api/v4/users?username=<name>` (документировано, поиск регистронезависим). Ответ — JSON **массив** (0 или 1 элемент), НЕ единичный объект; на промах отдаёт пустой массив `[]`, а не 404. Numeric id — поле `id` на верхнем уровне элемента.
- Пустой массив → ошибка «user not found», выполнение останавливается.
- Статус реализации: резолюция username → numeric id сегодня **НЕ кешируется**. `gitlab.ResolveUserID` не имеет ни одного вызывающего кода в репозитории (кроме собственного теста), и ни один тип в `cache/` не хеширует username — кешировать пока попросту нечему. Кеширование результата резолюции остаётся не выполненной пока задачей, как и кеширование smoke-теста (раздел 5.5): появиться оно должно на стороне вызывающего слоя (`bogoslav-cli`/`bogoslav-mcp`) или как отдельная функция `cache/`.
- Резолвленный numeric ID — то, что пишется в `query.user_id` артефакта-1 и с чем сравнивается `notes[].author.id`.
- `GET /users/:id/events` по докам принимает в путь «ID **или** Username» (колонка типа — integer, но описание параметра — «ID or Username of a user»; поведение подтверждено вживую, описание выигрывает у колонки типа).
- Требуемый scope токена для events: `read_user` или `api` (см. §2.5).

### 5.1. Стратегия events (основная)

1. Кандидаты: `GET /users/:id/events?action=commented&after=<from-1d>&before=<to+1d>&per_page=100`.
2. Форма события-комментария (из research): top-level `project_id`, `action_name: "commented on"`, `target_type: "Note"` (или `DiscussionNote`), вложенный объект `note` с полями `system`, `noteable_id`, `noteable_type`, `noteable_iid`. Одно событие = один комментарий.
3. Маппинг события на MR: пара `(project_id, note.noteable_iid)`. На клиенте оставляем только события с `note.noteable_type == "MergeRequest"` и `note.system == false`.
4. Предварительный счётчик на MR строится из событий; MR с предварительным счётчиком ≥ N становятся кандидатами — это **надмножество (superset)**, не финальный фильтр: точную границу `> N` пиннит только точный подсчёт (раздел 5.4).
5. Точный подсчёт по каждому кандидату — раздел 5.4. В артефакт-1 идёт только точное число.
6. Фильтр по группе/репозиторию: набор проектов группы — `GET /groups/:id/projects?include_subgroups=true`; события чужих проектов отбрасываются на клиенте.

### 5.2. Стратегия bruteforce (фолбэк)

1. Перебор списка MR по каждому проекту из области поиска (проект задан явно, либо все проекты группы через `GET /groups/:id/projects?include_subgroups=true`).
2. **Предикат окна: `created_before=to` + `updated_after=from`. `updated_before` добавлять НЕЛЬЗЯ** — он отрежет MR, обновлённые после окна.
3. Пре-фильтр: пропускать MR с `user_notes_count < N` — это верхняя граница числа комментариев любого одного пользователя, тоже **надмножественная (superset)** оценка, не финальный фильтр. Пре-фильтр требует дефолтного view; **`view=simple` использовать нельзя** (в нём нет `user_notes_count`).
4. Даты в MR API инклюзивны (из доков: "Returns merge requests created on or after the given date and time").
5. По каждому выжившему MR — точный подсчёт (раздел 5.4).
6. Документированные отказы, которые клиент обязан обрабатывать: `408` при таймауте запроса в БД GitLab (ретрай с бэкоффом), `429` при использовании параметра `search` (параметр `search` в переборе не использовать).
7. **`non_archived=false` отправляется ЯВНО на обоих спискующих MR-эндпоинтах, где параметр вообще документирован** — `GET /merge_requests` и `GET /groups/:id/merge_requests` (`gitlab.MergeRequests`, `gitlab.GroupMergeRequests`, константа `nonArchivedFalse`). Их дефолты на GitLab 18.11 РАЗНЫЕ (группа: `true`, глобальный список: `false` — раздел 2.0, п. 3): не полагаться на дефолт, слать значение явно на обоих, иначе bruteforce по группе тихо теряет MR из заархивированных проектов там, где events всё равно видит их через комментарии. `GET /projects/:id/merge_requests` (`gitlab.ProjectMergeRequests`, единственный явно заданный проект) параметр `non_archived` не документирует вообще и потому не отправляет его.

### 5.3. Автоселектор

Падать на bruteforce, если выполняется хотя бы одно:

- (а) начало диапазона (`from`) старше 3 лет от текущей даты — retention событий 3 года ("GitLab removes user activity events older than 3 years from the events table for performance reasons");
- (б) smoke-тест DiscussionNote на инстансе провалился ИЛИ дал неубедительный результат (`failed` или `unknown`, раздел 5.5);
- (в) включён строгий режим (`--strict`).

Уточнение к (а) для целевого GitLab 18.11 (раздел 2.0, п. 2; сверено с `18-11-stable-ee`): удаление старых событий выполняет `PruneOldEventsWorker` — `CUTOFF_DATE = 3.years + 1.day`, `DELETE_LIMIT = 10_000` записей за один прогон, cron `0 */6 * * *`, под ops-флагом `ops_prune_old_events` (`default_enabled: true`, добавлен в 15.10, MR !115100). Флаг выключается только через Rails-консоль администратором инстанса, UI-настройки нет, а само окно в 3 года не параметризуется на стороне GitLab. Отсюда следует, что 3 года — это НЕ гарантированный пол (флаг могут выключить, и события проживут дольше) и НЕ жёсткий потолок (на загруженном инстансе `DELETE_LIMIT=10_000`/6ч может не успевать вычищать всё вовремя, и события старше порога транзиторно переживают его). Наш выбор — считать «диапазон старше retention» сигналом для bruteforce — остаётся безопасным дефолтом именно потому, что он ошибается в сторону более медленной, но корректной стратегии в обе стороны этого отклонения, а не полагается на точность порога.

Выбранная стратегия фиксируется в артефакте-1 (`query.strategy`).

`autoselect.go` вычисляет отсечку retention через `opts.now().UTC()`, прежде чем извлекать год/месяц/день: без `.UTC()` дата отсечки текла бы по локальному часовому поясу процесса и могла разойтись на день с UTC-датами `from`/`to` (раздел 4.1 задаёт их в UTC). Тот же приём (`.UTC()` перед извлечением Y/M/D) применён и в `gitlab.SmokeTest` для окна смоук-теста (раздел 5.5).

### 5.4. Точный подсчёт (общий для обеих стратегий)

По `GET /projects/:id/merge_requests/:iid/discussions`: считать записи в `notes[]`, где `author.id == user`, `system == false`, `created_at` внутри диапазона `[from, to]`. `from`/`to` — даты `YYYY-MM-DD`, раскрываются в инклюзивный диапазон UTC-моментов `[from T00:00:00Z, to T23:59:59.999Z]`; `created_at` заметки сравнивается с этим диапазоном напрямую. Пагинация: дефолт 20, `per_page` до 100 — всегда запрашивать `per_page=100`.

Считать по `/notes` **запрещено**: цитата доков — "Items of type `DiscussionNote` are not returned as part of the Note API". `/notes` молча теряет ответы в тредах.

### 5.5. Smoke-тест DiscussionNote

Причина (из research): доки противоречат сами себе — Discussions API утверждает, что `DiscussionNote` в Events API недоступен вообще, а issue #466426 показывает, что события приходят, если не фильтровать по типу. Противоречие апстримом не разрешено, поэтому надёжность events-стратегии проверяется на конкретном инстансе.

Процедура выполняется `gitlab.Client.SmokeTest`:

1. Взять события `action=commented` пользователя за недавнее окно (дефолт: последние 90 дней — константа `smokeWindowDays`, без фильтра `target_type`).
2. Выбрать до 5 MR-кандидатов из этих событий; для каждого получить `/discussions` и посчитать точное число комментариев юзера в том же окне.
3. Если по `/discussions` у юзера есть комментарии-ответы в тредах, а суммарное число событий по этому MR меньше числа комментариев по `/discussions` — события теряют `DiscussionNote`: **тест провален**, автоселектор уходит в bruteforce.
4. Если в выборке не нашлось ни одного треда с ответами юзера — результат «неизвестно» (`unknown`). Решение — **консервативное**: автоселектор уходит в bruteforce, а не в events. Причина: если events на этом инстансе молча теряет `DiscussionNote`, предварительный счётчик **недоучивает** — MR не станет кандидатом и точный пересчёт по `/discussions` на нём вообще не запустится, то есть это тихий false negative, который механизм точного пересчёта не ловит (он лечит только перезачёт кандидатов, не их пропуск). Факт «smoke-тест неубедителен» пишется в лог и в провенанс артефакта-1 (поле `query.smoke: passed|failed|unknown`).

Статус реализации: `SmokeTest` сам по себе не кеширует свой результат — каждый вызов заново делает round-trip `events` + до 5×`discussions`. Persist-кеш «на инстанс+юзера с тем же TTL» из более ранней редакции этого раздела ещё не построен ни в `cache/`, ни в `gitlab/`; когда появится вызывающий слой (`bogoslav-cli`/`bogoslav-mcp`, раздел 7), кеширование смоук-теста — на его стороне или как отдельная функция `cache/` — остаётся открытой работой, не выполненной пока задачей этой ревизии.

### 5.6. Ловушки — обязательные требования (все из research, нарушать нельзя)

1. **Запрещено `target_type=note`** в запросе событий: этот фильтр молча отсекает `DiscussionNote` (ответы в тредах) и даёт заниженные счётчики без ошибок. Фильтровать только `action=commented`, тип проверять на клиенте по `note.noteable_type == "MergeRequest"`.
2. **Запрещён подсчёт по `/notes`** — только `/discussions` (см. 5.4).
3. **Retention 3 года**: диапазон с `from` старше 3 лет обязан автоматически падать на bruteforce (см. 5.3а).
4. **Smoke-тест обязателен** при настройке подключения / первом запуске events-стратегии на инстансе (см. 5.5).
5. **Инклюзивность `after`/`before` Events API не документирована**: запрашивать с паддингом ±1 день (`after = from - 1d`, `before = to + 1d`) и точно фильтровать на клиенте по `created_at` события.
6. **`updated_before` в переборе запрещён** (см. 5.2.2).
7. **Пре-фильтр `user_notes_count` требует дефолтного view**, не `view=simple` (см. 5.2.3).
8. **Фильтров `commented_by` (и любых фильтров по автору комментария) в списках MR не существует** — не пытаться «оптимизировать» стратегию несуществующими параметрами. Также не существуют: заметки по автору across projects в GraphQL, метрики комментариев в Contribution Analytics, фильтры по автору/дате на `/notes` и `/discussions`, keyset-пагинация для MR/notes/discussions/events.

### 5.7. Границы пакета `search/` — вход уже резолвлен для GitLab, не для человека

`search.Params{UserID int64; Range domain.DateRange; MoreThan int; Scope{GroupID, ProjectID *gitlab.ID}}` принимает `gitlab.ID` — новый тип (`gitlab/id.go`), который несёт **либо** уже резолвленный numeric id (`gitlab.NumericID(int64)`), **либо** namespaced-путь как есть (`gitlab.PathID(string)`, например `"my-group/my-project"`). Это резолюция «для GitLab», не «для человека»: GitLab 18.11 документирует каждый `:id` path-параметр как «The ID or URL-encoded path» (раздел 14, п. 1 — вопрос закрыт), так что `--group`/`--project`-путь идёт в `Scope` напрямую, без отдельного шага строка→numeric-id. Единственное место, где всё же нужен настоящий numeric id, — client-side фильтр events-стратегии по полю `project_id` события (оно всегда числовое, раздел 5.1.2): для этого случая `scopeProjectSet` (`search/events.go`) резолвит path-project через один вызов `client.GetProject` (см. ниже). `search/` по-прежнему не резолвит `--user` (username или id, раздел 5.0) — это остаётся обязанностью вызывающих слоёв (`bogoslav-cli`/`bogoslav-mcp`) до вызова `search.Find`.

`search/` не делает HTTP-запросов сам: он объявляет собственный потребительский интерфейс `search.Client` (методы `CommentEvents`, `Discussions`, `GetProject`, `GroupProjects`, `MergeRequests`/`GroupMergeRequests`/`ProjectMergeRequests`, `ProjectMergeRequestsByIIDs`, `SmokeTest`) и проверяет его реализацией через `var _ Client = (*gitlab.Client)(nil)` на этапе компиляции, не называя конкретный тип `*gitlab.Client` ни в одной сигнатуре стратегии — это то, что делает стратегии тестируемыми на фейковом клиенте (раздел 2.4).

Дополнительно зафиксировано текущей реализацией:

- `search.DefaultRetentionYears = 3` — дефолт для `Options.RetentionYears` (раздел 5.3а), настраиваемый параметр, а не захардкоженная константа автоселектора.
- **Обогащение кандидатов events-стратегии** (`search/enrich.go`, `enrichEventsCandidates`) закрывает бывшее расхождение формы результата между стратегиями (раздел 4.2, раздел 14 п. 11): вызывается из `Find` ТОЛЬКО для events-стратегии и ТОЛЬКО после того, как `resolveCandidates` применил `> MoreThan` — то есть только к выжившим кандидатам, никогда ко всему их набору. Группирует выживших по `ProjectID` и делает ОДИН вызов `client.ProjectMergeRequestsByIIDs` на каждый отдельный проект (`iids[]`, пачками по 100 — `gitlab.iidBatchSize`). `iids[]` документирован ТОЛЬКО у проектного списка MR на 18.11 — ни у группового, ни у глобального его нет, — поэтому пакетирование принципиально одно на проект, никогда одно на весь поиск. Выживший кандидат без совпавшей строки обогащения (например, MR удалён после того, как комментарий был оставлен) сохраняет свои `ProjectID`/`IID`/`CommentCount`, а не выбрасывается из результата. Регрессионный тест — `search.TestFind_eventsAndBruteforceStrategiesProduceEquivalentMergeRequests`.
- **`gitlab.GetProject(ctx, project gitlab.ID) (domain.Project, error)`** — `GET /projects/:id` (18.11: «id | integer or string | Yes | The ID or URL-encoded path of the project»), возвращает `gitlab.ErrProjectNotFound` на 404. `scopeProjectSet` (`search/events.go`) резолвит non-numeric project scope РОВНО ОДИН РАЗ за поиск через этот вызов — никогда внутри цикла по событиям или по MR-кандидатам (тест: `search.TestFind_eventsStrategyWithPathProjectScopeResolvesOnce`). Group-scope-по-пути в этом резолве не нуждается: `client.GroupProjects` всегда возвращает numeric id независимо от того, как задана сама группа.
- Граница `> more_than` применяется **ровно в одном месте** — `resolveCandidates` (вызывается из `search.Find`), после точного подсчёта по `/discussions`. Оба кандидатных пре-фильтра (события: предварительный счётчик `>= more_than`; bruteforce: `user_notes_count >= more_than`, т.е. пропускаются только строго `< more_than`) — надмножественная (superset) оценка, не финальный фильтр.
- Бисекция под-окна (раздел 6, п. 7) **чистая (без пересечений)** для списка событий (`splitDateRange`) — каждое событие несёт один `created_at`, поэтому деление на `[from, mid]`/`[mid+1, to]` — точное разбиение без дублей и пропусков. Для списка MR (`splitMergeRequestWindow`) бисекция **пересекающаяся по замыслу**: асимметричный предикат (`created_before` + `updated_after`, без `updated_before`) означает, что MR, созданный до `mid` и обновлённый после `mid`, попадает в обе половины — поэтому после слияния результатов **обязательна** дедупликация по `(project_id, mr_iid)` (`dedupeMergeRequests`).
- `splitMergeRequestWindow` отказывается делить окно, протяжённость которого меньше 2 дней, и возвращает `search.ErrWindowNotSplittable` (тот же сигнал, что и `splitDateRange` на однодневном диапазоне событий). Причина: при протяжённости ровно 1 день старая арифметика `midDate` (`from + days/2`, где `days/2 == 0` при `days == 1`) делала второе под-окно ИДЕНТИЧНЫМ входному — рекурсия не сходилась к базовому случаю, то есть окно продолжало бы бесконечно дробиться на само себя и бомбардировать API. `splitDateRange` (события) в этой же точке безопасен без изменений: при протяжённости 1 день он даёт два строго меньших диапазона (см. выше).

## 6. Rate limiting и пагинация — требования

Всё реализуется в `gitlab/`:

1. **Константу лимита не зашивать.** gitlab.com: 2000 запросов/мин на аутентифицированного пользователя. Self-managed: троттлинг выключен по умолчанию; при включении дефолт 7200 запросов за 3600 с (~2 req/s). Лимитер настраивается динамически по заголовкам.
2. **Лимитер решает по каждому ответу, а не по пути эндпоинта.** Читать заголовки лимитов на каждом ответе: `RateLimit-Limit`, `RateLimit-Name`, `RateLimit-Observed`, `RateLimit-Remaining`, `RateLimit-Reset`. Если они присутствуют — клиент адаптируется, замедляясь заранее по мере приближения `RateLimit-Remaining` к нулю, не дожидаясь 429.
3. **На 429 соблюдать `Retry-After`** (на 429 также приходит `RateLimit-ResetTime`). Ждать указанное время, затем повторить запрос — правило действует независимо от наличия остальных заголовков лимита.
4. **Если ответ (любой путь, включая 429) не несёт заголовков лимита — фолбэк на экспоненциальный бэкофф.** Решение принимается по факту заголовков в конкретном ответе, не «зашивается» за конкретным путём эндпоинта. Причина: research подтверждает, что у Projects/Groups/Users API 429 на gitlab.com не несут информационных заголовков, но остаётся невыясненным, попадает ли туда же `/users/:id/events` — эндпоинт основной events-стратегии (раздел 5.1). Лимитер, зашитый на путь, рисковал бы никогда не адаптироваться именно на этом горячем пути; ответ-ориентированная логика снимает эту неопределённость.
5. Лимит 60/мин на заметки касается только их **создания**; чтение — под общим лимитом. Мы ничего не создаём, но знание фиксируется, чтобы реализатор не закладывал лишний троттлинг.
6. **Пагинация**: только offset (keyset для нужных нам ресурсов не существует). `per_page` максимум 100 — всегда использовать 100.
7. **Порог 10k**: заголовки `x-total`/`x-total-pages` опускаются, если запрос вернул бы >10000 записей. Поэтому длинные диапазоны **режутся на под-окна по датам, а не листаются вглубь**. Механика: если при листании окна достигнут порог страниц, соответствующий 10000 записей (100 страниц по 100), окно делится пополам по дате и обе половины обрабатываются отдельно, рекурсивно.
   - **Под-окна — приём пагинации по API, а не партиция результата.** Для списка MR (bruteforce, предикат `created_before` + `updated_after`, раздел 5.2) деление `[from,to]` по `mid` даёт окно A (`created_before=mid`, `updated_after=from`) и окно B (`created_before=to`, `updated_after=mid`). MR, созданный до `mid` и обновлённый после `mid`, попадает в **оба** окна — дубликат. MR с `created_at == updated_at == mid` может не попасть **ни в одно**, если `updated_after` эксклюзивен. Поэтому после слияния результатов под-окон **обязательна дедупликация** по `(project_id, mr_iid)`.
   - Для `/users/:id/events` (единственный таймстамп `created_at` на событие) бисекция чистая — дублей и пропусков не даёт, дедупликация не нужна. Асимметричный предикат MR-списка — причина, по которой bruteforce-бисекция нечистая, а events-бисекция чистая.
   - Критерий приёмки — раздел 12, п. 25.
8. `408` (таймаут БД) — ретрай с экспоненциальным бэкоффом, ограниченное число попыток (дефолт 3, настраивается: `gitlab.WithMaxAttempts`, дефолт `defaultMaxAttempts = 3`).
9. Порог 10k реализован ровно в `gitlab.paginate`: `perPage = 100`, `maxPage = 100` (`100×100 = 10 000`); если 100 полных страниц пройдены без единой короткой страницы, `paginate` останавливается (глубже не листает) и возвращает всё собранное вместе с `gitlab.ErrPageLimitReached` — саму бисекцию окна выполняет вызывающий код в `search/` (раздел 5.7), не `gitlab/`.
10. Datetime-параметры двух эндпоинтов форматируются по-разному, оба намеренно:
    - Список MR (`created_before`/`updated_after`, раздел 5.2.2): `t.UTC().Truncate(time.Second).Format(time.RFC3339)` — целые секунды, вида `2019-03-15T08:00:00Z`, единственная документированная форма для этих параметров ([`docs.gitlab.com/api/merge_requests/`](https://docs.gitlab.com/api/merge_requests/)). Более ранняя реализация использовала `RFC3339Nano` (дробные секунды) — это было дефектом, исправлено; дробные секунды для этих двух параметров недокументированы, поэтому явно не отправляются.
    - Events API `after`/`before` (раздел 5.1.1): `YYYY-MM-DD`, дата без времени — тип параметра на стороне GitLab задан как `date`, не datetime (подтверждено исходником GitLab, `lib/api/helpers/events_helpers.rb`: `optional :before, type: Date`). Паддинг ±1 день плюс точная клиентская фильтрация по `created_at` (раздел 5.6.5) компенсируют недокументированную инклюзивность границ.
    - Ответные таймстампы (`created_at`, `updated_at`, ...): GitLab непоследователен даже внутри одного документированного примера — `created_at` без дробной части, `merged_at` с миллисекундами. Оба варианта парсятся: `time.Time`'s `UnmarshalJSON` через `encoding/json` уже принимает и целые, и дробные секунды под `RFC3339` — это стандартное поведение стандартной библиотеки, а не отдельный код в этом проекте; проверено тестом `TestClient_MergeRequests_parsesBothFractionalAndWholeSecondTimestamps`.

Реализованы только кирпичи, которые тулы будут оборачивать: internal-пакеты `artifact/`, `cache/`, `search/`, `filter/`, `classify/`, `stats/` (разделы 4–6, 8) — все с тестами, все зелёные (`go test ./...`).

### 7.1. SDK и транспорт

- Официальный `github.com/modelcontextprotocol/go-sdk`, **пин на v1.6.x** (актуальная стабильная — v1.6.1). Пре-релизы v1.7.0-pre.* не использовать.
- Транспорт — **stdio** (`StdioTransport`): все целевые инструменты спавнят MCP-сервер локально. HTTP+SSE не использовать — транспорт устарел в ревизии спеки 2025-03-26.
- Ревизия протокола MCP: 2025-11-25 (её полностью покрывает v1.6.x).

### 7.2. Тулы

Каждый тул — тонкая обёртка над internal-пакетами; вход и выход каждого тула описаны JSON-схемами, которые входят в `contracts/openapi.yaml` (раздел 10). Все тулы, пишущие артефакты, принимают `format` (`text|yaml|json|html`), `out` (каталог), `refresh` (bool) и возвращают путь к артефакту + краткую сводку.

| Тул | Назначение | Вход (ключевое) | Выход |
|---|---|---|---|
| `find_mrs` | Артефакт-1: MR, где юзер оставил > N комментариев (или точечный режим — один MR) | `user`, `from`, `to`, `more_than`, `group?`, `project?`, `mr?`, `strict?` | путь к `mr_list`, сводка: число MR, стратегия, число запросов |
| `get_comments` | Артефакт-2: комментарии юзера | `from_artifact?` (путь к `mr_list`) ИЛИ явный список `{project, mr_iid}`, `user`, `from?`, `to?` | путь к `comment_list`, число комментариев |
| `get_classify_batch` | Выдать агенту батч на разметку | путь к `comment_list`, `model`, `taxonomy_version?` | батч комментариев + таксономия + JSON-схема результата + шаблон промпта; либо `cached: true` + путь к готовому артефакту-3 |
| `save_labels` | Принять разметку, провалидировать, записать артефакт-3 | путь к `comment_list`, разметка (массив `{note_id, label}`), блок `classifier` (tool, model, taxonomy_version, classified_at) | путь к `labeled_comments` ИЛИ ошибка валидации со списком нарушений |
| `filter_comments` | Артефакт-4: фильтрация по меткам и прочим фильтрам | путь к `labeled_comments`, `labels[]`, `from?`, `to?`, `group?`, `project?` | путь к `filtered_comments`, число записей |
| `get_stats` | Агрегаты поверх любого артефакта | путь к артефакту | сводка (распределение по MR / меткам / датам) — без записи файла, если не передан `out` |

Точечный режим `find_mrs`: если передан `mr` (iid) вместе с `project`, тул возвращает `mr_list` с единственным элементом — этим MR (без поиска кандидатов events/bruteforce), точный подсчёт всё равно идёт через `/discussions` (раздел 5.4).

#### 7.2.1. Выходной контракт `get_stats`

`get_stats` — агрегат по содержимому одного входного артефакта (любого `kind`), Go-тип со следующими полями (сериализуется как обычный артефакт-ответ, без `schema_version`/`source`/`query`, т.к. не пишет новый артефакт-кеш):

```yaml
source_kind: mr_list          # kind входного артефакта: mr_list | comment_list | labeled_comments | filtered_comments
total_items: 42                # общее число записей во входном артефакте
by_mr:                         # только для comment_list | labeled_comments | filtered_comments
  - project_id: 123
    mr_iid: 77
    count: 8
by_label:                      # только для labeled_comments | filtered_comments
  bug: 5
  style: 3
by_date:                       # счётчик записей по дневным бакетам created_at (YYYY-MM-DD)
  "2026-03-01": 4
  "2026-03-02": 1
```

Для `mr_list` заполняются только `source_kind`/`total_items` (по числу `items`); `by_mr`/`by_label`/`by_date` — пустые/отсутствуют, т.к. в `mr_list` нет отдельных записей-комментариев с датами и метками.

### 7.3. Соответствие CLI ↔ MCP

Дерево команд `bogoslav-cli` зеркалит тулы: `find`, `comments`, `classify prepare` (= `get_classify_batch`), `classify save` (= `save_labels`), `filter`, `stats`. Общие флаги: `--user`, `--from`, `--to`, `--group`, `--project`, `--mr`, `--more-than`, `--strict`, `--format`, `--out`, `--cache-ttl`, `--refresh`. Одна функция internal-пакета на пару команда+тул.

## 8. Семантическая разметка

Статус реализации: пакет `classify/` (таксономия, шаблон промпта, JSON Schema, валидатор, провенанс — разделы 8.3–8.5) полностью реализован и покрыт тестами. Тулы `get_classify_batch`/`save_labels`, которые связывают `classify/` с `artifact/`/`cache/` поверх MCP-транспорта, ещё не написаны (раздел 7) — весь этот раздел описывает контракт, который они обязаны реализовать, а не то, что уже вызывается по сети.

### 8.1. Принцип

Размечает **вызывающий агент**, не MCP и не CLI. MCP владеет контрактом:

- `get_classify_batch` отдаёт: батч комментариев, таксономию (версионированную), JSON-схему результата, шаблон промпта (всё из `classify/`);
- агент (скилл диктует, каким сабагентом/моделью) выполняет разметку;
- `save_labels` валидирует пришедшую разметку по JSON-схеме и пишет артефакт-3. Невалидная разметка в артефакт не попадает — тул возвращает ошибку со списком нарушений, файл не создаётся.

### 8.2. Выбор модели

Модель/сабагент выбирается **на уровне скилла** (текст SKILL.md указывает агенту, кого спавнить). Там, где у инструмента нет сабагентов, — деградация: разметка выполняется основной моделью инлайном. MCP не ВЫБИРАЕТ модель и никогда её не вызывает; идентификатор модели передаётся тулу (`get_classify_batch`) на вход и фиксируется — для провенанса (раздел 8.3) и как часть ключа кеша разметки (раздел 8.4).

### 8.3. Провенанс

Артефакт-3 обязан нести блок провенанса (значения передаёт агент в `save_labels`):

```yaml
classifier:
  tool: opencode
  model: glm-5.2
  taxonomy_version: 3
  classified_at: 2026-07-15T16:40:00Z
```

### 8.4. Кеш разметки

Батч с неизменившимися комментариями (хеш по `items` артефакта-2) и той же парой `model` + `taxonomy_version` повторно не гоняется: `get_classify_batch` обнаруживает существующий артефакт-3 с совпадающим хешем и парой и возвращает `cached: true` + путь, не выдавая батч.

### 8.5. Таксономия

Живёт в `classify/` как версионированные данные (файл в пакете), поле `taxonomy_version` — целое, пользователь может её редактировать. Дефолтный набор v1 (для комментариев код-ревью), поставляется «из коробки»:

`bug`, `style`, `naming`, `architecture`, `performance`, `security`, `test`, `docs`, `question`, `nitpick`, `praise`, `other`.

`other` — обязательный фолбэк-лейбл: у валидатора всегда должна быть корректная цель, даже если ни одна содержательная метка не подошла. Это не только соглашение — оно принудительно проверяется кодом: `classify.NewTaxonomy` и `Taxonomy.UnmarshalJSON` отклоняют набор меток без `"other"` ошибкой `classify.ErrTaxonomyMissingOther`, включая загрузку пользовательской таксономии из файла. Изменение состава меток (добавление/удаление/переименование) обязано увеличивать `taxonomy_version` — это делает провенанс артефакта-3 однозначным. Схема валидации: каждая запись разметки — `{note_id, label}`, где `label` ∈ множеству меток таксономии; каждый `note_id` из батча размечен ровно один раз. `classify.Validate` собирает **все** нарушения за один проход (плохая метка, лишний `note_id`, пропущенный `note_id`, повторный `note_id`), не останавливаясь на первом — вызывающий код (`save_labels`) получает исчерпывающий список сразу.

## 10. Контракты: `contracts/openapi.yaml`

Статус реализации: `contracts/` сегодня содержит только `.gitkeep` — ни `openapi.yaml`, ни команда `go generate` ещё не написаны. Что подтверждено рабочим кодом уже сейчас: `classify.ResultSchema()` вызывает `jsonschema.For[[]NoteLabel](&jsonschema.ForOptions{})` (`apps/internal/classify/schema.go`) и тестами (`TestResultSchema_isDraft2020_12`, `TestResultSchema_marshalJSONRoundTrip`) проверено, что (а) схема не проставляет `$schema` и тем не менее резолвится и валидирует под драфтом 2020-12 (например, отклоняет лишнее свойство через `additionalProperties: false`), и (б) схема раунд-трипится через `MarshalJSON`/`UnmarshalJSON` без потерь. Это подтверждает описанный ниже механизм генерации схемы из Go-типа для одного конкретного типа (`classify.NoteLabel`); связка «одна генерация — два потребителя» (openapi.yaml из `contracts/` и `inputSchema` тулов MCP-сервера) остаётся спроектированной, но не подключённой: `mcp.AddTool` нигде не вызывается, поскольку `bogoslav-mcp` — по-прежнему заглушка (раздел 7).

- Формат: **OpenAPI 3.1** (использует JSON Schema 2020-12, поэтому документ валиден и открывается в Redoc/Swagger).
- `paths` — пустой объект: REST API не делаем.
- В `components/schemas` лежат: схемы всех четырёх артефактов (`mr_list`, `comment_list`, `labeled_comments`, `filtered_comments`) и схемы входа/выхода **каждого** MCP-тула из раздела 7.2.
- **Источник истины — Go-типы** (в `domain/` и типы входов/выходов тулов). Из них генерируется JSON Schema; она же идёт в `inputSchema` MCP-тулов — по спеке MCP `inputSchema` это и есть JSON Schema. Одна генерация — два потребителя (openapi.yaml и регистрация тулов), ручной дубликат схем запрещён.
- Генерация — команда `go generate` в модуле. Библиотека генерации JSON Schema из Go-типов: **`github.com/google/jsonschema-go/jsonschema`** (v0.4.3) — та же библиотека, которую `mcp.AddTool[In, Out]` использует внутри себя для вывода `inputSchema` из типа `In` через рефлексию (`jsonschema.ForType(rt, &jsonschema.ForOptions{})`), когда `Tool.InputSchema == nil`. Эмитируемый драфт — **JSON Schema 2020-12**, то есть ровно то, что принимает `components/schemas` OpenAPI 3.1; ключ `$schema` не проставляется. Описания свойств берутся из тега `jsonschema:"..."`.
- `*Server` не даёт публичного геттера зарегистрированных схем тулов — `AddTool` копирует `Tool`, структура вызывающего кода так и остаётся с `InputSchema: nil`; узнать схему в рантайме можно только через `tools/list`. Поэтому схема для `openapi.yaml` генерируется **отдельным вызовом той же библиотеки** на билд-тайме: `jsonschema.For[T](&jsonschema.ForOptions{})` (публичный generic; `ForType` — вариант с `reflect.Type`) на входных/выходных типах тулов, сериализация через `Schema.MarshalJSON()` в `components/schemas`. `mcp.AddTool` выводит схему из тех же Go-типов тем же кодовым путём библиотеки — идентичность схем гарантирована совпадением пути генерации, а не сверкой в рантайме.
- Важно: подпакета `go-sdk/jsonschema` в v1.6.1 **не существует** (удалён после v0.2.0) — использовать именно `github.com/google/jsonschema-go/jsonschema`. `For`/`ForType` в v0.4.x принимают аргумент `*ForOptions`. Правила вывода схемы структур: структура → `type: object` с запретом `additionalProperties`; свойства берутся из экспортируемых полей по JSON-имени; поля с `omitempty`/`omitzero` — опциональны, все остальные — обязательны (`required`).
- Поверхность CLI в OpenAPI **не описывается** — справка CLI генерируется в markdown из дерева команд cobra (та же генерация, что кормит SKILL.md, раздел 9.1).

## 14. Открытые вопросы / ТРЕБУЕТ УТОЧНЕНИЯ

1. ~~Резолюция строкового пути группы/проекта в numeric id~~ — **ЗАКРЫТО для request path-параметров.** Прежний открытый вопрос («принимает ли GitLab путь вместо numeric id в `:id`?») сверен с 18.11 доками напрямую: каждый `:id` path-параметр документирован как «The ID or URL-encoded path» (пример из доков: `GET /api/v4/projects/diaspora%2Fdiaspora`, с явным предупреждением URL-кодировать `/` как `%2F`). Новый тип `gitlab.ID` (`gitlab.NumericID(int64)` / `gitlab.PathID(string)`, `gitlab/id.go`) реализует это: `GroupProjects`, `Discussions`, `GroupMergeRequests`, `ProjectMergeRequests`, `GetProject`, `ProjectMergeRequestsByIIDs` принимают `gitlab.ID` напрямую, а `search.Params.Scope` несёт `*gitlab.ID` вместо `*int64` (раздел 5.7) — отдельный резолв-шаг строка→numeric-id для этих вызовов более не нужен. Операционная оговорка, задокументированная и в доках 18.11, и в `gitlab/id.go`: прокси перед GitLab, который декодирует `%2F` обратно в `/`, ломает путь-как-id 404-ой.
   Смежный узкий случай тоже закрыт: события несут `project_id` ВСЕГДА числовым, так что path-form project scope нужно резолвить один раз для client-side фильтра events-стратегии — это делает `search.scopeProjectSet` через новый `gitlab.GetProject` (раздел 5.7). Group-scope-по-пути в резолюции не нуждается вообще: `GroupProjects` всегда возвращает numeric id.
2. **Удаляется ли событие при удалении заметки** — апстримом не документировано. Влияет на предварительные счётчики events-стратегии. Митигция уже в дизайне: финальное число всегда пересчитывается по `/discussions`; вопрос остаётся для полноты кандидатов. Отразить в README (раздел «честность цифр»).
3. **Версия GitLab, где ИЗНАЧАЛЬНО ввели 3-летнее удаление событий, всё ещё не выяснена** (issue #408467 обсуждал 5 лет). Механика удаления на целевой 18.11 теперь задокументирована подробно (раздел 2.0 п. 2, раздел 5.3): `PruneOldEventsWorker`, `CUTOFF_DATE = 3.years + 1.day`, ops-флаг `ops_prune_old_events` (`default_enabled: true`, добавлен в 15.10, MR !115100) — но эта дата про флаг ВКЛЮЧЕНИЯ/ВЫКЛЮЧЕНИЯ удаления, не про версию, где появился сам 3-летний `CUTOFF_DATE`. Порог «3 года» в автоселекторе остаётся настраиваемым параметром (`search.Options.RetentionYears`, дефолт `search.DefaultRetentionYears = 3`, раздел 5.7) и безопасным дефолтом независимо от того, выяснится ли точная версия.
4. **Доступность `events(filter:)` в GraphQL на старых self-managed** — не проверена. GraphQL в этом ТЗ не используется, вопрос актуален только если появится GraphQL-маршрут.
5. **Библиотека JSONC-aware мёржа для Go** — выбрать при реализации (раздел 9.3.1); приёмка фиксирует поведение, не библиотеку.
6. **Подкоманды `kilo mcp`** (существует ли `add`) — не проверено в research; установщик для Kilo работает через файл `kilo.jsonc`, CLI-путь не использовать без проверки.
7. **Путь `cline_mcp_settings.json` на стороне IDE-расширения** — не проверен; таргетим документированный `~/.cline/mcp.json`.
8. ~~`cache.HeaderReader` не имеет ни одной боевой реализации~~ — **ЗАКРЫТО** (см. также раздел 4.6). `artifact.HeaderStore.FetchedAt(path string) (time.Time, error)` (`artifact/header_store.go`) теперь удовлетворяет интерфейс; компайл-тайм доказательство — `var _ cache.HeaderReader = (*HeaderStore)(nil)` в `artifact/header_store_test.go`, без цикла импортов. Остающийся, отдельный факт (не тот же вопрос, что был здесь): `cache.Lookup` всё ещё не вызывается ни одним путём кода продукта — не из-за отсутствующего имплементора, а потому, что `bogoslav-cli`/`bogoslav-mcp` остаются заглушками (раздел 7) и никто пока не собрал `cache.Lookup` + `artifact.HeaderStore` в одну цепочку. Это уже не вопрос этого пункта, а обычное следствие незаписанных тулов/команд (см. раздел 12).
9. **Конвертера `domain.Query` → `search.Params` всё ещё не существует** — это остаётся открытым, но опасность ключа кеша, привязанная к этому пункту, ПЕРЕФОРМУЛИРОВАНА (см. также раздел 4.6): раньше `search.Params.Scope` требовал уже резолвленные ЧИСЛОВЫЕ id при том, что `cache.QueryHash` хеширует `Group`/`Project` как СТРОКИ — рассинхрон ключа и области поиска был реальной дырой. Теперь `Scope{GroupID, ProjectID *gitlab.ID}` (раздел 5.7) принимает путь напрямую — путь и есть то, чем реально ходят в API (раздел 14, п. 1) — так что старое рассогласование «путь vs numeric id» закрыто. Другая, УЖЕ УЗКАЯ опасность остаётся и принимается как обоснованный риск, а не как нерешённая проблема: если группу/проект переименуют и НОВЫЙ объект займёт СТАРЫЙ путь, тот же ключ кеша однажды укажет на другие данные — но окно ограничено 24-часовым TTL. Волна CLI/MCP по-прежнему обязана написать сам конвертер `domain.Query` → `search.Params` — эта ревизия его не пишет.
10. **Конвертера `search.Result` → `artifact.MRList` всё ещё не существует** (см. также раздел 4.2) — это единственный оставшийся шов; пп. 11 и 12 прежних редакций закрылись на уровне `domain`/`gitlab`, но данные всё равно некому положить в `artifact.MRItem`, пока этого конвертера нет. Он также остаётся местом, где должно закрыться решение по п. 9 (как ключевать кеш).
11. ~~Стратегия `events` заполняет кандидатов MR только `ProjectID`/`IID`, расходясь по форме с `bruteforce`~~ — **ЗАКРЫТО.** Новая `search/enrich.go`'s `enrichEventsCandidates` дозаполняет `Title`/`WebURL`/`CreatedAt`/`UpdatedAt`/`ProjectPath` у выживших events-кандидатов (после `> MoreThan`, никогда до) через новый `gitlab.ProjectMergeRequestsByIIDs` — ОДИН вызов на проект, батчами по 100 `iids[]`. `Find` вызывает это только для events-стратегии; регрессия — `search.TestFind_eventsAndBruteforceStrategiesProduceEquivalentMergeRequests`. Подтверждено: эндпоинт для этого обогащения в кодовой базе ЕСТЬ (`iids[]` документирован только на `GET /projects/:id/merge_requests`, не на групповом и не на глобальном списке) — прежняя оговорка «эндпоинт не проверен» снята.
12. ~~У `domain.MergeRequest.ProjectPath` нет продюсера~~ — **ЗАКРЫТО на уровне `domain`/`gitlab`, ЧАСТИЧНО на уровне артефакта.** `gitlab.populateProjectPaths` заполняет `domain.MergeRequest.ProjectPath` из `References.full` (новый саб-тип `domain.References{Short, Relative, Full}` + метод `References.ProjectPath()`, делящий `Full` по последнему `!`) на каждом MR-list-эндпоинте, включая обогащение из п. 11 — без единого лишнего запроса. Тест: `gitlab.TestClient_MergeRequestLists_populateProjectPathFromReferences`. НО: `artifact.MRItem.ProjectPath` (поле самого артефакта-1) по-прежнему заполняют только тесты напрямую (`artifact/mr_list_test.go`, `filter/mr_test.go`, `stats/mr_list_test.go`) — этот пробел не самостоятельный, он тот же конвертер `search.Result` → `artifact.MRList` из п. 10, и закроется вместе с ним.
